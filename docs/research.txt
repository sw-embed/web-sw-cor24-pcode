Historically, how Pascal was bootstrapped
The earliest Pascal compiler was not originally bootstrapped in assembler in the way many people assume. Niklaus Wirth’s first FORTRAN-based attempt was unsuccessful; the successful implementation was done in a C-like language called Scallop and then translated by hand into Pascal for bootstrapping. Many later Pascal compilers became self-hosting. A famous exception is GNU Pascal, which was written in C. Turbo Pascal, much later, was famously written in assembly for speed and size. 

So the historical pattern is roughly:

prototype in another language

get a working compiler

rewrite or re-express the compiler in Pascal

use the previous compiler to compile the new compiler

That is exactly the path I would recommend for your ISA as well.

What to target first: native code or p-code?
For your situation, I would start with a Pascal-S / Pascal-P style subset and a p-code-like intermediate machine.

Why:

your ISA is 24-bit, which means you will be making many nonstandard representation decisions anyway

Historically, Pascal-P was created specifically as a compiler porting kit: Pascal source compiled to p-code for a virtual stack machine, then interpreted on each target. That was done to make Pascal portable. 

For a custom ISA, that strategy is still excellent.

Short answer on Pascal memory management
Classic Pascal was not generally garbage-collected. The usual model was:

stack allocation for locals, parameters, temporaries

static/global storage for globals

heap allocation for dynamically created structures via new

explicit reclamation, usually via dispose, or sometimes effectively “free everything at program end”

That is still the cleanest model for your first implementation. Standard Pascal and mainstream descendants historically center on explicit heap management rather than GC. Free Pascal’s docs and wiki still document New/Dispose and GetMem/FreeMem style management, and its heap debugging support tracks allocation/deallocation rather than automatic tracing GC. 

So for your first p-code runtime, I would recommend:

no GC initially

explicit heap operations

optional region/arena reset at process end

later, optional reference counting or tracing GC as a runtime extension for other languages

What should heap allocation look like?
For Pascal first version: explicit allocation
I recommend these VM/runtime operations:

HEAP_ALLOC size -> ptr

HEAP_FREE ptr

HEAP_ZERO ptr, size

optional HEAP_REALLOC only if another language later wants it

Pascal front end then maps:

new(p) → allocate size of pointed-to type, assign pointer

dispose(p) → free and optionally nil the pointer

That gives you a small, understandable runtime.

Should the heap be malloc/free style?
Yes, for the first version.

Even if the source language says new/dispose, under the hood that is basically a typed wrapper around an allocator API equivalent to malloc/free. Free Pascal’s documentation explicitly distinguishes Pascal-level New/Dispose from lower-level GetMem/FreeMem, which supports this layered interpretation. 

So the layering can be:

VM allocator: raw blocks

Pascal runtime: typed heap ops on top

That is clean.

One key decision: stack machine vs explicit locals model
Since this is p-code, the natural design is a stack-oriented assembler.

Example style:

proc main
    push 72
    call putc
    push 105
    call putc
    push 10
    call putc
    halt
endproc
But for debugging and maintainability, I recommend you also support named local slots or explicit frame offsets.

For example:

proc blink locals 1
loop:
    push 1
    out led
    call delay
    push 0
    out led
    call delay
    jmp loop
endproc
or, more machine-like:

proc blink 1
loop:
    push 1
    storeg LED_PORT
    call delay
    push 0
    storeg LED_PORT
    call delay
    jmp loop
endproc
That is enough.

Calling convention recommendation
Since this is for bring-up and testing, make the convention brutally simple.

Suggested convention
arguments are pushed on the eval stack

call foo

callee uses enter nlocals

arguments become addressable relative to frame pointer

return value is pushed on the eval stack before ret

caller removes args if needed, or callee does — pick one and stick to it

For simplicity, I would choose:

Better v1 convention
caller pushes args

call foo

callee reads args at known negative offsets from fp

callee leaves one return value on the eval stack, or none for procedures

callee pops its frame only

arguments are cleaned by callee or encoded in ret n

A nice compact form is:

ret 0 for no args

ret 2 meaning pop 2 args and return

or retv 2 meaning return value + pop 2 args

But that may complicate the VM slightly.

A simpler alternative:

keep call frames separate from eval stack

copy args into frame slots on call

That is cleaner but a bit more machinery.

For early work, either is fine. If you expect lots of debugging, a separate frame area is easier to reason about.

For your use case, I would recommend two abstractions for I/O
Since you want UART and LED access, decide whether p-code talks to:

Option A — VM-level abstract devices
Example:

sys LED_WRITE

Option B — memory-mapped addresses

Pros

Cons

Best answer: support both
Use memory-mapped addresses in the runtime if you want, but also provide small VM/syscall abstractions for common tasks.

For example:

putc implemented as runtime p-asm

inside it, either call sys PUTC on host, or do memory-mapped UART on target

That keeps the assembler small and the runtime portable.

One subtle point: typed vs untyped
You said “not support typing per se.”

That is right for the assembler source language. But the VM still needs operational widths.

Even in an “untyped” p-code assembler, you will still need distinctions like:

word load/store

byte load/store

signed vs unsigned compare, maybe

address vs integer conventions, even if both are same-sized

So the source language can be untyped, but the instruction set should still be explicit enough to avoid ambiguity.

For example:

load = word

loadb = byte

lt = signed compare

ltu = unsigned compare if needed later

This will matter for UART, chars, string traversal, and array/string runtime work.
